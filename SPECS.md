# 收纳物品地图 - 产品规格文档


## 1. 产品概述

### 一句话描述

帮助收纳师快速制作交互式物品地图，帮助住户随时查找家中物品位置。

### 解决的问题

收纳师完成上门整理后，需要制作"物品地图"交付给客户（标注每个柜子每一层放了什么）。当前做法是拍照后用PPT/PS手动标注，**每次耗时2-4小时**。我们将这个过程缩短到30分钟以内，并且交付物从静态图片升级为可搜索、可交互的数字地图。

### 用户角色

| 角色 | 描述 | 付费 |
|------|------|------|
| **收纳师**（Pro用户） | 创建和编辑物品地图，管理多个客户 | 付费订阅 |
| **住户**（客户端） | 查看/搜索/编辑物品地图 | 永久免费 |
| **个人用户** | 自己记录自己的家（无收纳师参与） | 免费（有限额） |

### 核心使用场景

```
收纳师上门 → 整理完一个柜子 → 打开App → 拍照 → 输入物品描述
→ 完成所有柜子后 → 一键生成分享链接 → 发给客户
→ 客户打开链接 → 搜索"护照" → "书架第三层"
```

## 2. 技术架构

### 2.1 平台选型：微信小程序

| 维度 | 决策理由 |
|------|---------|
| **开发语言** | JavaScript/TypeScript |
| **框架** | Taro（React语法编写小程序，一套代码可同时编译为H5） |
| **相机** | `wx.chooseMedia()` 拍照/选图，API完善 |
| **分享** | 微信内天然支持小程序卡片分享，零摩擦 |
| **用户触达** | 中国用户在微信中直接打开，无需安装 |
| **Demo展示** | 评委扫码即可体验 |

后续扩展：Taro可同时编译为H5/React Native，未来可拓展到其他平台。iOS原生版本留待产品验证后开发（届时可集成Apple RoomPlan/LiDAR扫描）。

注意：小程序没有DOM API（无document/window），依赖DOM的npm包（如Three.js）不能在小程序中直接使用，需要通过WebView嵌入H5页面来实现。

### 2.2 后端：微信云开发

一站式后端，零配置，和小程序深度集成。

| 服务 | 用途 | 免费额度 |
|------|------|---------|
| **云数据库**（MongoDB） | 所有结构化数据（Space/Room/Container/Slot） | 2GB |
| **云存储**（对象存储） | 照片文件 | 5GB |
| **云函数** | 后端逻辑（分享链接生成、权限校验） | 足够 |

- 照片处理：上传前在前端用canvas压缩到800px宽度、JPEG 80%质量，每张约100-200KB
- 结构化数据和照片都存储在云端，收纳师和住户通过同一个后端访问同样的数据

### 2.3 小程序版本与Demo展示

微信小程序有三种版本，**Hackathon用"体验版"即可，不需要正式审核上线**：

| 版本 | 谁能用 | 需要审核 |
|------|--------|---------|
| 开发版 | 开发者扫码 | 不需要 |
| **体验版** | 最多100个指定体验者 | **不需要** |
| 正式版 | 所有微信用户 | 需要（1-3天） |

操作步骤：
1. 注册微信小程序账号（个人类型，免费）→ mp.weixin.qq.com
2. 开发完成后在微信开发者工具中点"上传"
3. 在后台设为"体验版" → 生成体验版二维码
4. 评委扫码即可体验

团队协作：在后台添加团队成员为"开发者"，即可在各自手机上预览。

注意：个人类型小程序**不能使用 `<WebView>` 组件**。3D功能需要企业类型小程序，或通过分享链接在手机浏览器中打开，或用 `three-platformize` 库在小程序内原生渲染。

### 2.4 3D可视化（可选，作为亮点）

通过小程序的 `<WebView>` 组件嵌入一个Three.js H5页面，实现Plugin式扩展：

```
小程序（2D编辑/搜索/拍照）
  ↓ 点击"3D视图"按钮
<WebView src="https://domain.com/3d?spaceId=xxx" />
  ↓
Three.js H5页面（3D渲染，从同一个云数据库读取数据）
```

- 2D和3D完全解耦，3D是独立的H5页面
- 可以直接复用已有的Three.js代码（`buildRoomFromJSON`函数）
- 开发代价：约2-4小时（主要是数据对接）
- 不影响核心功能开发，可以在最后阶段添加

### 2.5 前端组件架构

```

### 2.6 权限与共享

```

## 3. 数据模型

### 3.1 实体关系

```
Space（一个家）
  ├── name: "张先生的家"
  ├── owner: User（收纳师）
  ├── sharedWith: User[]（住户，editor权限）
  └── rooms: Room[]

Room（房间）
  ├── name: "卧室" | "客厅" | "衣帽间" | "厨房"
  └── containers: Container[]（该房间内的所有容器）

Container（容器 — 核心对象）
  ├── id: uuid
  ├── name: "主衣柜" | "收纳箱A"
  ├── type: "wardrobe" | "cabinet" | "shelf" | "dresser" | "box" | "drawer_unit" | "custom"
  ├── movable: boolean
  │     false → 固定家具（衣柜、书架、电视柜）
  │     true  → 可移动收纳单元（收纳箱、Fits抽屉柜）
  ├── photo: url（容器整体的实拍照片）
  ├── slots: Slot[]（容器内部的分层/分区）
  └── children: Container[]（嵌套的可移动子容器）

Slot（分区/格子）
  ├── label: "第二层" | "上层抽屉" | "挂衣区"
  ├── type: "shelf" | "drawer" | "rod" | "open"
  ├── items: string（物品描述文字）
  └── photo: url（该层的照片）
```

### 3.2 Container嵌套

Container支持最多两层嵌套：固定家具 > 可移动收纳单元 > Slot

```
主衣柜 (Container, movable=false)
  ├── 顶层 (Slot)
  │   ├── 收纳箱A (Container, movable=true)
  │   │   └── 内部 (Slot): "夏季空调被×2"
  │   └── 收纳箱B (Container, movable=true)
  │       └── 内部 (Slot): "换季衣物"
  ├── 挂衣区 (Slot): "外套×10, 衬衫×8"
  └── 底层 (Slot)
      └── Fits抽屉柜 (Container, movable=true)
          ├── 第1层抽屉 (Slot): "内衣"
          ├── 第2层抽屉 (Slot): "袜子"
          └── 第3层抽屉 (Slot): "打底衫"
```

### 3.3 可移动容器的整体迁移

当用户将一个可移动容器从卧室搬到客厅时，只需修改其 `parentId`。该Container及其所有Slots和children整体迁移，物品记录完全不变。用户不需要逐层移动。

### 3.4 录入粒度

物品描述按**区域/类别汇总**，不逐件记录：

```
推荐: "春夏T恤×15、短裤×8"
推荐: "护肤品约20件（日常护肤套装+面膜库存）"
不推荐: "优衣库白色圆领T恤(XL) 1件、MUJI灰色V领T恤(L) 1件、..."
```

配合一张照片，用户查找时看照片即可辨认具体物品。

## 4. 商业模式

### 4.1 定价策略

```
免费版（个人用户）：
  - 1个Space，5个Container
  - 基本搜索
  - 照片存储100MB

Pro版（收纳师）¥29/月：
  - 无限Space，无限Container
  - 模糊搜索 + 同义词
  - 照片存储5GB
  - 客户管理面板
  - 品牌水印定制
  - 数据导出

住户端：永久免费
  - 收纳师分享的Space永久可查看/编辑/搜索
```

### 4.2 为什么按Space而不是按Container收费

按Container收费（按量计费）会导致收纳师在录入时"舍不得多加容器"，为了省钱而跳过一些柜子或简化录入。这会直接降低物品地图的完整性和数据质量，从而降低住户端的体验——住户搜不到东西就不会继续使用，收纳师也就失去了"物品地图"这个差异化卖点。

**固定订阅制（无限Container）鼓励收纳师尽可能详细地录入**，数据越完整→住户体验越好→口碑越好→收纳师越愿意续费。

### 4.3 增长飞轮

```
收纳师使用App → 制作物品地图分享给客户
→ 客户体验好 → 推荐给朋友
→ 朋友也请收纳师（或自己使用免费版）
→ 更多收纳师注册使用
```

### 4.4 市场规模

- 中国家政/收纳服务市场持续增长
- 职业收纳师群体快速扩大
- 收纳服务客单价 ¥3,000-15,000/单，用户有支付能力
- 1万名付费收纳师 × ¥348/年 = **¥348万年营收**

## 5. Hackathon计划（48小时）

### 5.1 开发流程

每个团队成员都使用AI agent（Claude Code等）进行全栈开发。不按传统前端/后端分工，而是**按功能模块垂直拆分，最大化并行**。

```
阶段0：共同基础（所有人一起，30分钟）
  → 初始化项目 + 确定数据模型 + 约定接口
  
阶段1：并行开发（每人一个独立模块）
  → 各模块互不阻塞，用Mock数据独立开发
  
阶段2：集成 + Demo准备
  → 串联页面 + 联调数据 + 统一UI + 填充测试数据
```

### 5.2 模块拆分

模块之间唯一的接口是**数据库的数据结构**（阶段0约定好）。各模块可完全并行开发。

| 模块 | 内容 | 依赖 |
|------|------|------|
| **A. 数据录入** | Space/Room/Container/Slot 的创建、编辑、删除全链路 + 云数据库CRUD | 无（核心模块，工作量最大） |
| **B. 拍照 + 照片** | 拍照/选图 + 前端压缩 + 云存储上传 + 照片关联 | 作为组件嵌入模块A |
| **C. 搜索 + 分享** | 全局搜索 + 结果展示 + 分享链接生成 + 住户视图 | 只需从数据库读取，与A完全并行 |
| **D. 3D可视化**（可选） | Three.js H5页面 + 从数据库生成3D场景 | 完全独立 |


**降低阻塞的关键**：
- 阶段0把数据结构100%定下来
- 各模块用本地Mock数据开发，不等其他模块完成
- 每个模块一个Git分支，最后合并

### 5.3 Demo要点

比赛形式：评委自由到各团队电脑前互动体验。准备以下展示要素：

- **问题展示**：收纳师当前花2-4小时用PPT做物品地图的痛点
- **核心流程**：创建家→添加容器→输入物品→拍照→搜索定位→分享给客户
- **评委互动**：让评委扫码在自己手机上体验搜索功能
- **商业模式**：收纳师订阅制，住户免费
- **未来方向**：3D可视化、AI语义搜索、LiDAR自动建模
